# Video Lens 技术栈文档

## 1. 项目目标

`Video Lens` 的核心目标不是“做一个模型实验脚本”，而是“做一个普通人也能直接操作的视频分析工具”。

这个目标直接决定了技术选型：

- 要足够轻，部署简单
- 要能本地运行，不强迫用户上云
- 要支持视频文件输入，而不是只接受文本
- 要把视频分析流程做成可视化界面
- 要兼顾单视频和批量文件夹处理

## 2. 总体架构

项目采用非常直接的三层结构：

1. 前端展示层
2. Python 后端服务层
3. 模型与视频处理层

整体流程如下：

```text
用户选择视频/文件夹
        ↓
浏览器把文件上传到本地 Python 服务
        ↓
Python 解析任务参数并创建 job
        ↓
根据 provider 进入不同处理路径
        ↓
Ollama 路线：ffprobe/ffmpeg → 抽帧 → 调用 Ollama
Gemini 路线：直接上传视频文件 → 调用 Gemini Files API
        ↓
后端保存每个视频的进度、日志、结果
        ↓
前端轮询 job 状态并渲染队列和结果
```

## 3. 技术栈总览

### 后端

- `Python`

- `threading`
- `subprocess`
- `urllib.request`

### 前端

- `HTML`

### 视频处理

- `ffmpeg`

### 模型接入

- `Ollama API`
- `Gemini Files API`

## 4. 为什么后端选 Python

Python 的优势在这个项目里非常明确：

- 调用系统命令非常方便
- 处理文件上传、JSON、线程这些任务足够直接
- 非常适合把 `ffmpeg`、`ffprobe`、本地模型 API 串起来
- 对于这种中小型工具，开发速度快，维护成本低

我们没有使用更重的后端框架，比如 `Django` 或 `FastAPI`，原因不是它们不好，而是这个项目暂时不需要它们。

本项目当前需求是：

- 启动一个本地服务
- 接收文件上传
- 维护任务状态
- 调用本地或云端模型

这些事情用 Python 标准库已经足够完成。

这样做的好处是：

- 依赖更少
- 克隆后更容易跑起来
- 对初学者更友好
- 整个项目更容易读懂

## 5. 为什么前端选原生 HTML / CSS / JS

这个项目的前端不是复杂的业务后台，也不是大型 SPA。

它本质上是一个单页工具面板，前端主要负责：

- 表单输入
- 任务提交
- 状态轮询
- 队列渲染
- 结果展示

因此使用原生前端的优势很明显：

- 不需要 Node 打包环境
- 不需要 React / Vue 构建链
- 减少安装和版本问题
- 部署简单，后端直接静态托管即可

这也是为什么项目对“只有少量代码基础的人”更友好。

他不需要先理解：

- bundler
- npm scripts
- Vite
- component state 管理
- 构建产物

直接打开源码就能理解页面结构、样式和交互。

## 6. 为什么要用 ffmpeg / ffprobe

视频分析不是单纯把文件交给模型就结束了。

后端首先需要知道：

- 视频时长
- 分辨率
- 编码信息
- 是否包含音频

这部分信息由 `ffprobe` 获取。

如果走本地 `Ollama` 路线，还需要把视频转成视觉模型更容易处理的输入形式，也就是关键帧序列。

这部分由 `ffmpeg` 完成。

选择 `ffmpeg` / `ffprobe` 的原因：

- 业界事实标准
- 稳定
- 跨平台
- 几乎所有视频格式都能处理
- 非常适合自动化管道

## 7. 为什么接入 Ollama

`Ollama` 的价值在于把本地模型运行这件事变简单。

对于本项目来说，它的优势包括：

- 用户可以在自己电脑上跑模型
- 本地隐私更好
- 提供统一的本地 HTTP API
- 用户不需要手写复杂推理脚本
- 模型切换成本较低

我们当前默认使用：

- `qwen3-vl:8b`

原因是它处在一个比较实用的平衡点：

- 有真实视觉理解能力
- 本地部署可行
- 质量和资源占用之间比较平衡

## 8. 为什么还要接入 Gemini

虽然本地模型很重要，但很多用户会问两个问题：

1. 有没有“直接上传视频文件”的方案？
2. 有没有不需要在本地自己处理帧的方案？

所以项目保留了 `Gemini` 路线。

这样做有两个价值：

- 产品层面给用户一个“直接传视频”的选择
- 技术层面让项目同时覆盖本地方案和云端方案

因此项目不是只押注一种路线，而是把现实里最常见的两种能力都接起来：

- 本地可控
- 云端直接

## 9. 为什么批量分析采用顺序处理

文件夹批量分析是本项目的重要功能。

但批量并不等于一开始就要并发。

我们当前采用顺序处理，原因很现实：

- 本地模型本来就吃资源
- 顺序执行更稳定
- 更容易定位失败的视频
- 日志更清晰
- 用户更容易理解当前系统在处理哪一个文件

对于工具型产品，稳定性通常比“理论最高吞吐”更重要。

后续如果要优化吞吐，可以再考虑：

- 有限并发
- 可配置 worker 数
- 失败重试

但第一版先把“稳定、清楚、好用”做好。

## 10. 后端开发思路

### 第一步：先统一任务模型

不管是单视频还是文件夹批量，后端都需要一个统一的任务对象。

因此后端设计了 `Job` 概念：

- `job_id`
- 当前状态
- 日志
- items 列表
- 最终结果
- 错误信息

这样前端轮询时就不需要区分太多情况。

### 第二步：把单视频和批量视频统一成 items

单视频其实可以看作“只有一个 item 的批量任务”。

这样带来的好处是：

- 后端逻辑统一
- 前端队列统一
- 状态统计统一

这是一个很关键的简化点。

### 第三步：把 provider 差异收口在 pipeline 层

真正不同的不是任务系统，而是分析管道：

- `run_ollama_pipeline`
- `run_gemini_pipeline`

这样做的优势是：

- 任务调度层保持稳定
- 将来新增 provider 会更容易
- 更方便排查某一条模型链路的问题

### 第四步：让前端只做展示，不做复杂业务判断

前端只负责：

- 提交任务
- 轮询 job
- 渲染状态
- 渲染结果

业务事实全部以后端 job 数据为准。

这能避免前端和后端各自维护一套复杂状态机。

## 11. 前端开发思路

### 设计目标

不是做一个普通表单页，而是做一个“审片工作台”。

所以前端结构上拆成了几个清晰区域：

- Hero 头部：说明产品定位
- Control Deck：配置任务
- Mission Control：看实时进度
- Contact Sheet：看批量队列
- Inspector：看当前视频详情

### 为什么这样拆

因为视频分析和普通聊天框不同，它天然更适合“工作台”式布局。

用户通常需要同时关注：

- 现在在跑什么
- 一共有多少视频
- 哪些完成了
- 哪条失败了
- 当前看的是哪一条结果

所以单栏布局不适合这个产品。

### 为什么强调队列 + 详情

批量分析里，用户最常见的操作不是“只看总结果”，而是：

- 先扫一眼全体状态
- 再点进某一条视频看详情

因此我们采用：

- 左边列表
- 右边详情

这比把所有结果直接堆在一页里更实用。

## 12. 本地视频分析路线的开发思路

### Ollama 路线

流程是：

1. 接收视频文件
2. `ffprobe` 读取元数据
3. 根据时长、目标 FPS、最大帧数计算抽帧频率
4. `ffmpeg` 抽取关键帧
5. 把帧转成 base64 图片数组
6. 调用 `Ollama /api/generate`
7. 保存结果并回传前端

这样设计的原因是：

- 对用户来说仍然是“直接选视频”
- 对模型来说仍然是“按视觉帧理解”
- 对工程实现来说最稳定

### Gemini 路线

流程是：

1. 接收视频文件
2. 上传到 Gemini Files API
3. 轮询直到文件可用
4. 调用 `generateContent`
5. 返回结果
6. 尝试删除临时远程文件

这样设计的优点是：

- 用户视角真的是直接传视频
- 项目同时具备本地与云端两种产品能力

## 13. 当前技术栈的优势

### 对用户

- 更容易运行
- 不需要理解太多工程概念
- 本地和云端两种方案都能选
- 可以直接分析单个视频或整个文件夹

### 对开发者

- 依赖少
- 架构简单
- 代码容易读
- 易于扩展 provider
- 易于定位问题

### 对项目演进

- 可以继续加导出功能
- 可以继续加更多模型
- 可以继续加并发队列
- 可以继续加数据库和历史任务

## 14. 当前技术栈的取舍

这套技术栈不是“最现代的堆料组合”，而是“当前阶段最务实的组合”。

我们明确做了这些取舍：

- 不上 React / Vue，换来更低环境门槛
- 不上 FastAPI，换来更少依赖和更简单启动
- 不先做数据库，换来更快实现和更低复杂度
- 不先做并发队列，换来更稳定的批量执行

这不是能力问题，而是产品阶段选择。

## 15. 将来可扩展的方向

如果项目继续发展，下一步技术演进可以考虑：

- `FastAPI`：更适合更复杂的 API 结构
- `SQLite / PostgreSQL`：保存历史任务和结果
- `WebSocket`：替代轮询，做更实时的状态更新
- 并发任务调度器：提升批量吞吐
- 结果导出：Markdown、JSON、CSV
- 用户配置文件：保存默认 provider、prompt、参数

## 16. 总结

`Video Lens` 的技术栈不是为了炫技，而是为了完成一个清晰目标：

把“视频分析”这件事，做成一个真实能用、普通用户也能跑通的本地工具。

所以这套栈的核心优势不是“前沿”，而是：

- 清楚
- 够用
- 易跑
- 易扩展
- 符合这个项目的实际阶段
